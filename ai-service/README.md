# AI Module Planning

---

## 주요 담당 업무

### 1. 프로젝트 메타데이터 자동 분석
- 프로젝트 설명문, README, 코드 파일을 분석하여 핵심 정보를 추출
- 기술 스택, 프로젝트 주제, 핵심 키워드, 사용 언어 등을 구조화
- 추출된 메타데이터를 검색 및 추천 기능에 활용할 수 있도록 정리

### 2. 자연어 기반 검색
- 사용자의 자연어 질문에서 의도와 조건을 해석
- 질문 속 도메인, 기술 스택, 목적, 난이도 등의 정보를 추출
- 추출된 조건을 기반으로 관련 프로젝트를 검색할 수 있는 구조 설계

### 3. 추천 기능
- 검색 결과 및 메타데이터를 바탕으로 유사 프로젝트 추천
- 공통 기술 스택, 유사 주제, 난이도 등을 기준으로 추천 결과 생성
- 추천 이유를 함께 제공할 수 있도록 출력 구조 설계

### 4. 시맨틱 검색 최적화
- 프로젝트 설명 및 문서를 임베딩으로 변환
- 벡터 기반 유사도 검색 구조를 검토
- 키워드 검색과 의미 기반 검색을 결합한 탐색 방식 설계

### 5. 백엔드 연동
- AI 분석 결과를 백엔드 서비스에서 활용할 수 있도록 연동 구조 검토
- 검색 결과, 추천 결과, 메타데이터를 API 형태로 제공할 수 있도록 설계
- 백엔드 담당과 협업하여 서비스 적용 흐름 정리

## 자연어 질의 예시

다음은 본 서비스에서 처리하고자 하는 대표적인 자연어 질의 예시이다.

- 헬스케어 분야에서 이미지 기반 AI 프로젝트 예시 추천해줘
- React와 Spring을 같이 사용한 프로젝트 찾아줘
- 딥러닝 초보자가 참고하기 좋은 프로젝트 5개 추천해줘
- Unity랑 MediaPipe 조합으로 만든 선배 프로젝트 있나?
- 캡스톤 프로젝트 중에서 추천 시스템 관련 사례 보여줘

### 질의 해석 예시
| 자연어 질의 | 해석 조건 |
|---|---|
| 헬스케어 분야에서 이미지 기반 AI 프로젝트 예시 추천해줘 | 도메인=헬스케어, 유형=이미지 기반, 주제=AI |
| React와 Spring을 같이 사용한 프로젝트 찾아줘 | 기술 스택=React, Spring |
| 딥러닝 초보자가 참고하기 좋은 프로젝트 5개 추천해줘 | 난이도=초급, 분야=딥러닝, 개수=5 |
| Unity랑 MediaPipe 조합으로 만든 선배 프로젝트 있나? | 기술 스택=Unity, MediaPipe |
| 캡스톤 프로젝트 중에서 추천 시스템 관련 사례 보여줘 | 과제 유형=캡스톤, 주제=추천 시스템 |

# 메타데이터 입력 출처 정의

프로젝트의 핵심 정보를 정확히 추출하기 위해서는 단일 입력값이 아니라  
여러 자원을 함께 참고해야 한다고 판단하였다.

## 주요 입력 출처

| 입력 출처 | 설명 | 추출 가능한 정보 |
|---|---|---|
| 프로젝트명 | 등록 시 입력하는 프로젝트 제목 | 주제, 핵심 키워드 |
| 한 줄 소개 | 프로젝트의 짧은 요약 설명 | 목적, 분야 |
| 상세 설명 | 프로젝트 내용에 대한 본문 설명 | 도메인, 문제 정의, 목적, 기술 스택 |
| README | 프로젝트 설명 문서 | 구조, 기능, 기술 스택, 실행 방법 |
| 보고서/발표자료 | 프로젝트 결과물 문서 | 사용 기술, 개발 배경, 성과 |
| 코드 파일명 | 예: `app.py`, `Main.java` | 사용 언어, 구성 요소 힌트 |
| 폴더 구조 | 프로젝트 디렉터리 구성 | 아키텍처 구조, 모듈 분류 |
| 설정 파일 | `requirements.txt`, `package.json`, `pom.xml` 등 | 라이브러리, 프레임워크, 실행 환경 |

# 설계 시 고려 사항

2주차 작업을 진행하면서 다음과 같은 고려 사항을 정리하였다.

## 1. 단일 입력 출처만으로는 충분하지 않음
프로젝트 설명문이 짧거나 부정확할 수 있으므로,  
README, 보고서, 코드 구조를 함께 참고해야 한다.

## 2. 추출 결과의 정규화가 필요함
예를 들어 사용자가 다음과 같이 서로 다르게 적을 수 있다.

- JS / JavaScript
- Spring / Spring Boot
- AI / Artificial Intelligence

이러한 표현은 내부적으로 통일된 값으로 정규화해야  
검색과 추천 품질을 높일 수 있다.

## 3. 확정 정보와 추정 정보를 구분할 필요가 있음
일부 메타데이터는 문서에 명시적으로 존재하지만,  
일부는 코드 구조나 표현을 바탕으로 추정해야 한다.  
따라서 향후 구현 시에는 신뢰도 개념을 둘 수 있다고 판단하였다.

## 4. 검색 친화적인 구조가 필요함
메타데이터는 단순 저장용이 아니라  
키워드 검색, 필터링, 추천, 시맨틱 검색에 모두 활용되어야 한다.
